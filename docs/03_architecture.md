# D3 - FCF 总体架构图与数据流

## 1. 目的

本文档定义 FCF 系统的总体架构、模块边界、数据流、事件流、决策生命周期，以及实盘、影子、回放三种模式之间的关系。

D3 的作用不是写具体策略，而是确定系统骨架。

后续所有模块开发都必须遵守本文档定义的架构边界。

## 2. 架构原则

FCF 架构遵守以下原则：

- 先定义契约，再写业务逻辑
- 所有模块通过事件通信
- 决策提案不可直接执行
- 风控治理层拥有最终否决权
- 执行层只接收标准订单
- 实盘可以停止，影子学习不能停止
- 所有决策必须可以回放
- 硬熔断优先于智能判断

## 3. 总体分层

FCF 分为七层：

| 层级 | 名称 | 职责 |
|---|---|---|
| L1 | 数据接入层 | 收集外部原始数据 |
| L2 | 数据标准化层 | 把原始数据转换为统一格式 |
| L3 | 事件总线层 | 负责模块之间的事件传递 |
| L4 | 智能分析层 | 识别状态、生成特征、形成判断 |
| L5 | 决策提案层 | 生成标准化决策提案 |
| L6 | 风控治理层 | 审核、否决、限额、熔断 |
| L7 | 执行、审计与回放层 | 执行订单、记录事件、支持回放 |

## 5. 核心模块

| 模块 | 所在层级 | 说明 |
|---|---|---|
| data_ingestor | 数据接入层 | 接收盘口、赔率、新闻、伤病、天气、赛程等数据 |
| normalizer | 数据标准化层 | 把不同来源的数据转换为统一格式 |
| event_bus | 事件总线层 | 分发系统事件 |
| regime_radar | 智能分析层 | 判断当前市场或比赛状态 |
| feature_engine | 智能分析层 | 生成模型需要的特征 |
| model_engine | 智能分析层 | 运行统计模型或预测模型 |
| llm_reasoner | 智能分析层 | 负责解释、归纳、推理 |
| strategy_proposer | 决策提案层 | 生成标准决策提案 |
| policy_engine | 风控治理层 | 判断提案是否违反规则 |
| risk_guardian | 风控治理层 | 处理风险、冲突和熔断 |
| capital_manager | 风控治理层 | 控制资金、仓位、限额 |
| executor | 执行层 | 执行已经批准的标准订单 |
| event_store | 审计与回放层 | 保存所有事件 |
| replay_engine | 审计与回放层 | 根据历史事件重建系统状态 |
| shadow_simulator | 审计与回放层 | 模拟不真实下单的执行结果 |

## 6. 标准数据流

正常情况下，数据流如下：

1. 外部数据进入数据接入层。
2. 数据接入层只负责收集，不负责判断。
3. 原始数据进入标准化层。
4. 标准化层输出统一格式事件。
5. 事件进入事件总线。
6. 智能分析层从事件总线读取事件。
7. 智能分析层生成状态判断、特征和模型输出。
8. 决策提案层生成标准决策提案。
9. 风控治理层审核决策提案。
10. 审核通过后，提案变成标准执行订单。
11. 执行层执行订单。
12. 执行结果写入审计与回放层。
13. 回放系统可以重建当时的完整决策过程。

## 8. 实盘、影子、回放

FCF 同时支持三种运行方式。

### 8.1 实盘模式

实盘模式可以产生真实订单。

实盘模式必须满足：

- 系统状态允许实盘
- 没有触发硬熔断
- 风控审核通过
- 资金限制通过
- 没有同场冲突

### 8.2 影子模式

影子模式不产生真实订单。

影子模式用于：

- 熔断后的持续学习
- 策略验证
- 风险观察
- 模拟滑点和流动性影响
- 对比实盘与理论表现

当系统进入风险状态时，实盘可以停止，但影子模式继续运行。

### 8.3 回放模式

回放模式用于重建历史。

回放模式必须可以回答：

- 当时系统看到了什么数据？
- 当时生成了什么提案？
- 哪个规则批准或拒绝了提案？
- 如果进入影子模式，模拟结果如何？
- 如果重放同一组事件，结果是否一致？

## 10. 模块边界

### 10.1 数据层不能做决策

数据接入层和标准化层只处理数据，不判断是否下注，不判断是否执行。

### 10.2 智能层不能直接碰资金

智能分析层可以生成判断和提案，但不能直接执行真实订单。

### 10.3 风控层拥有否决权

任何决策提案，只要被风控治理层拒绝，就不能进入执行层。

### 10.4 执行层必须保持无状态

执行层只接收标准订单，不读取策略内部逻辑。

### 10.5 审计层必须完整记录

所有原始事件、标准化事件、提案、审核结果、执行结果都必须记录。
