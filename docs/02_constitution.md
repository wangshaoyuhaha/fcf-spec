# D2 - FCF Constitution

## 1. 目的

本文档定义 FCF 的系统宪法。

D2 的作用是确定系统的最高设计原则、生命周期、熔断规则、影子模式原则和风控优先级。

后续所有模块、策略、模型和执行逻辑，都不能违反本文档。

## 2. 核心原则

FCF 遵守以下核心原则：

1. Contract First  
先定义接口、事件和数据契约，再写业务逻辑。

2. Event Driven  
所有模块通过事件通信，不直接依赖彼此内部逻辑。

3. Immutable Decision  
决策提案一旦生成，不能被偷偷修改，只能追加审计记录。

4. Math by Code, Reasoning by LLM  
确定性计算交给代码，解释、归纳和推理交给大模型。

5. Money Stops, Learning Never Stops  
真实资金可以停止，影子学习不能停止。

6. Every Decision Replayable  
所有决策必须可以通过历史事件完整回放。

7. Every Layer Replaceable  
只要遵守契约，每一层都可以替换。

8. Stateless Execution  
执行层只接收标准订单，不读取策略内部逻辑。

9. Policy Above Intelligence  
风控治理层高于智能分析层，拥有最终否决权。

10. Evolution Before Optimization  
先保证系统可演进，再追求收益优化。

11. Radar Before Prediction  
先识别状态，再决定是否预测或出手。

12. Hard Circuit Breaker First  
硬熔断优先于软治理和智能判断。

## 3. 系统生命周期

FCF 的全局生命周期包括：

- BOOTING：系统启动中
- READY：系统准备完成
- LIVE：实盘运行中
- DEGRADED：系统降级
- SHADOW：影子模式
- RECOVERING：恢复中
- STOPPED：系统停止

生命周期方向：

BOOTING -> READY -> LIVE

LIVE 可以进入 DEGRADED。

DEGRADED 可以进入 SHADOW。

SHADOW 可以进入 RECOVERING。

RECOVERING 可以回到 LIVE。

任何严重风险都可以进入 STOPPED。

## 4. 硬熔断规则

硬熔断是最高优先级规则。

只要触发硬熔断，系统必须停止真实执行，并进入影子模式或停止状态。

第一版硬熔断条件：

- 连续 API 失败达到阈值
- 实时回撤超过账户允许范围
- 单笔滑点明显超过预估滑点
- 系统时钟漂移超过允许范围
- panic regime 与 tail risk 同时升高
- 数据质量严重下降
- 同一比赛或同一市场出现冲突订单
- 风控模块不可用

触发硬熔断后，系统必须发布事件：

fcf.circuit_breaker.triggered

## 5. 软治理规则

软治理不一定立即停止系统，但可以降低风险暴露。

软治理可以执行：

- 降低仓位
- 禁止新订单
- 只允许影子执行
- 降低模型权重
- 要求人工复核
- 延迟执行
- 提高置信度门槛

软治理由 policy_engine 和 risk_guardian 共同负责。

## 6. 影子模式原则

影子模式不影响真实资金。

影子模式用于：

- 熔断后的持续学习
- 验证策略是否仍然有效
- 观察模型表现
- 模拟滑点和流动性影响
- 对比实盘结果和理论结果
- 为恢复实盘提供证据

进入影子模式后，系统仍然需要记录完整事件链。

## 7. 滑点模拟原则

影子模式和回放模式不能假设理想成交。

滑点模拟需要考虑：

- 当前盘口价格
- 市场深度
- 订单大小
- 流动性冲击
- 估计成交价格
- 实际或模拟滑点

第一版简化规则：

模拟成交价 = 盘口价格 加减 流动性影响

流动性影响由订单大小、市场深度和影响系数决定。

## 8. 风控优先级

FCF 的优先级从高到低为：

1. 硬熔断
2. 风控治理
3. 资金限制
4. 数据质量
5. 决策提案
6. 智能推理
7. 收益优化

任何收益目标都不能高于风控规则。

## 9. 决策不可直接执行

智能分析层和策略提案层只能生成提案。

提案必须经过风控治理层审核。

只有审核通过后，才可以变成标准执行订单。
